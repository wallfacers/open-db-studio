## UI Object Protocol

UI Objects are stateful elements (editors, forms, canvases) identified by `type` + `objectId`. Use `target="active"` for the focused one. Mutations use **JSON Patch (RFC 6902)**; `[name=xxx]` addresses array elements by name.

### Four Core Tools

- **`ui_read(object, target?, mode?)`** — modes: `state` (current values) / `schema` (patchable paths, `x-addressable-by` hints) / `actions` (action list w/ param schemas) / `full` (all three, fewer round trips)
- **`ui_patch(object, target?, ops, reason?)`** — JSON Patch ops: `replace` / `add` (append to `/arr/-`, upserts by `name`) / `remove`. Addressing: `/columns[name=email]/dataType`
- **`ui_exec(object, target?, action, params?)`** — run an action (create / delete / batch / side-effects)
- **`ui_list(filter?)`** — **首选发现入口**：用户提"当前 / 打开的 / 这个 / 正在编辑的" UI 元素时先调用。Filter: `type` / `keyword` / `connectionId` / `database`

### Discovery-First

Before using a new object type: `ui_read(mode="actions")` to see actions, `ui_read(mode="schema")` to see patchable paths. **`ui_patch`** for simple field updates; **`ui_exec`** for creates / deletes / batch / side-effects. When in doubt, `ui_exec`.

### Actions by Object

- **workspace**: `open(params={type, connection_id?, database?, table?, job_id?, title?})` — `type` ∈ `query_editor` / `table_form` / `er_canvas` / `metric_form` / `new_migration_job` / `migration_job`; `close(target)`; `focus(target)`
- **query_editor**: `run_sql`, `format`, `undo`, `focus`, `set_context({connection_id, database, schema})`. Write SQL by patching `/content`.
- **table_form**: `preview_sql`, `save`, `add_foreign_key({column, referencedTable, referencedColumn})`, `remove_foreign_key({constraintName})`, `add_index({name, columns, type})`, `batch_create_table({tableName, columns, indexes})`
- **db_tree**: `search({keyword, type})`, `refresh`, `expand({nodeId})`, `select({nodeId})`, `locate_table({connection_id, database, table})` (auto-loads ancestors)
- **history**: `list`, `undo`

**Auto mode**: writes to `table_form` / `metric_form` require Auto ON; otherwise Adapter returns `{status:"error", message:"需要开启 Auto 模式"}` — tell the user to enable it.

## Table Design — `init_table_form`

**`init_table_form(connection_id, database, table_name, columns, indexes?, foreignKeys?, comment?, engine?="InnoDB", charset?="utf8mb4")`** — open & populate a table designer in one call. Prefer this over multi-step open+patch for new tables. **Do NOT use for ER diagrams — use `init_er_table` / `er_batch`.**

- `columns[]`: `{name, dataType, length?, isNullable?=true, defaultValue?, isPrimaryKey?=false, extra?, comment?}`
- `indexes[]`: `{name, columns[], unique?=false}`
- `foreignKeys[]`: `{column, referencedTable, referencedColumn, constraintName?, onDelete?="NO ACTION", onUpdate?="NO ACTION"}`

Example:
```
init_table_form(connection_id=1, database="app", table_name="orders",
  columns=[
    {name:"id", dataType:"BIGINT", isPrimaryKey:true, extra:"AUTO_INCREMENT"},
    {name:"user_id", dataType:"BIGINT", isNullable:false},
    {name:"amount", dataType:"DECIMAL", length:"10,2"}],
  indexes=[{name:"idx_user_id", columns:["user_id"]}])
```

For **existing** tables: `ui_exec(workspace, open, {type:"table_form", connection_id, database, table:"users"})`, then patch. Preview DDL with `ui_exec(table_form, preview_sql)`.

## ER Diagram — `er_canvas`

- **`init_er_table(target?="active", table_name, columns, indexes?, comment?, position?={x:100,y:100})`** — [DEPRECATED] single-table. Columns use **snake_case**: `{name, data_type, length?, nullable?, is_primary_key?, is_auto_increment?, is_unique?, comment?}`
- **`er_batch(target?="active", ops, returnState?, dryRun?)`** — multi-step with `$N.tableId` / `$N.columnMap.<col>` variable binding across steps

Actions (in `er_batch` or `ui_exec`): `batch_create_table`, `add/update/delete_table`, `add/update/delete_column`, `add/update/delete_relation`, `add/update/delete_index`, `replace_columns`, `replace_indexes`, `reorder_columns`. See `ui_read(er_canvas, mode="actions")` for schemas.

Example — two tables + relation:
```
er_batch(ops=[
  {action:"batch_create_table", params:{name:"users", columns:[
    {name:"id", data_type:"BIGINT", is_primary_key:true, is_auto_increment:true},
    {name:"email", data_type:"VARCHAR", length:255}]}},
  {action:"batch_create_table", params:{name:"orders", columns:[
    {name:"id", data_type:"BIGINT", is_primary_key:true, is_auto_increment:true},
    {name:"user_id", data_type:"BIGINT"}]}},
  {action:"add_relation", params:{
    source_table_id:"$0.tableId", source_column_id:"$0.columnMap.id",
    target_table_id:"$1.tableId", target_column_id:"$1.columnMap.user_id"}}])
```

Addressing: `[id=5]` or `[name=users]`. **Relations cannot be patched** — use `ui_exec` `add_relation` / `update_relation`. Single table → `init_er_table`; multi-table or relations → `er_batch`.

## Migration Job (MigrateQL)

1. **Open**:
   - New: `ui_exec(workspace, open, {type:"new_migration_job", title})` → `{objectId:"migration_job_<id>", job_id}` (objectId is stable across tab close/reopen)
   - Existing: `ui_exec(workspace, open, {type:"migration_job", job_id, title})`
2. **Read**: `ui_read(migration_job, mode="state")` → `{scriptText, jobId, name, status}`
3. **Write**: `ui_patch(migration_job, ops=[{op:"replace", path:"/scriptText", value:"..."}])`
4. **Actions**: `format`, `run` (saves & executes), `stop`

### MigrateQL Syntax

```
USE src = CONNECTION('source_mysql');
USE dst = CONNECTION('target_pg');

SET parallelism=8, read_batch=2000, write_batch=1024, byte_capacity=16M, error_limit=100;

MIGRATE FROM src.mydb.users INTO dst.appdb.user_accounts
  MAPPING (
    id         -> id,
    user_name  -> username :: VARCHAR(100),
    created_at -> create_time)
  WHERE status = 'active'
  ON CONFLICT UPSERT BY (id)
  INCREMENTAL ON updated_at
  CREATE IF NOT EXISTS;

MIGRATE FROM src.mydb.orders INTO dst.appdb.orders MAPPING (*) ON CONFLICT REPLACE;
```

- `MAPPING (src -> dst :: TYPE, ...)` or `(*)` for same-name auto-map
- `ON CONFLICT UPSERT|REPLACE|SKIP|INSERT|OVERWRITE BY (cols)`
- `WHERE <cond>`, `INCREMENTAL ON <col>`, `CREATE IF NOT EXISTS`, `SET k=v, ...`

### Batch Size Tuning

**Always query row count first**: `SELECT COUNT(*) FROM <source_table>`. Optionally `get_table_schema` for column width.

| Rows | parallelism | read_batch | write_batch | byte_capacity | Notes |
|---|---|---|---|---|---|
| <100K | 2 | 1000 | 512 | 4M | defaults fine |
| 100K–1M | 4 | 2000 | 1024 | 8M | range-split |
| 1M–10M | 8 | 5000 | 2048 | 16M | sweet spot |
| 10M–100M | 8 | 5000 | 2048 | 32M | add error_limit |
| >100M | 8 | 5000 | 2048 | 64M | chunk with WHERE |

**Defaults (optimized for 1M–10M)**: `parallelism=8, read_batch=2000, write_batch=1024, channel_capacity=8, byte_capacity=16M, max_bytes_per_tx=8M`.

**Byte literals** for `byte_capacity` / `max_bytes_per_tx`: `16M`, `1G`, etc. Units `K`/`M`/`G` (case-insensitive); raw ints OK. **Normalized to nearest power-of-2 MB** (1M, 2M, 4M, 8M, 16M, 32M, 64M, 128M, 256M, 512M, 1G, 2G, 4G), e.g. `15M→16M`, `9M→8M`. Max **4GB** (higher capped).

**Engine clamps**: `read_batch [1,50000]`, `write_batch [1,5000]`, `parallelism [1,16]`, `channel_capacity [1,64]`, `transaction_batch_size [1,100]`, `byte_capacity`/`max_bytes_per_tx` normalized pow2-MB ≤4GB (setting None disables backpressure — dangerous!).

**Rules**:
- `parallelism > 1` **requires integer PK** on source (else falls back to single-cursor OFFSET pagination)
- **Always set `byte_capacity`** — without it, in-flight data accumulates unbounded (38GB+ spikes observed)
- `>10M` rows: set `error_limit` (e.g. 100) so pipeline stops on bad data instead of silently skipping
- `>50M` rows: chunk with `WHERE` (date / ID range) rather than one job
- `read_batch > 5000` rarely helps — source DB is the bottleneck
- Memory formula: `byte_capacity + (parallelism × write_batch × row_size)` ≈ peak (tested: 10M rows × 33 cols ≈ 185 MB)

## Knowledge Graph

Stores FKs, inferred links, virtual relationships, business aliases, metric nodes. Use to resolve ambiguity before multi-table SQL.

- **`graph_query_context(connection_id, question)`** — **preferred entry** for multi-table joins / ambiguous names. Returns relevant tables, structured JOIN paths (cardinality, via fields, `sql_hint`), matched metrics
- `graph_search_tables(connection_id, question)` — fuzzy-search by business meaning (preferred over `list_tables`)
- `graph_find_join_paths(connection_id, from_table, to_table)` — shortest JOIN path with `sql_hint`
- `graph_get_ddl(connection_id, table_name)` — compact CREATE TABLE for SQL-gen context
- `graph_search_metrics(connection_id, keyword?, table_name?)` — metric nodes
- `graph_debug_links(connection_id, table_name?)` — diagnostic when `find_join_paths` returns empty

**Guidelines**:
- Multi-table: call `graph_query_context` first
- Check `context_quality`: `graph_hit` = high confidence, `fts_fallback` = approximate, `empty` / `schema_only` = fall back to DB read tools
- Use `join_paths[].sql_hint` directly — don't guess JOIN conditions
- Skip graph for single-table queries

## Virtual Relationships (Comment-Based)

When tables lack native FKs, relationships can be declared in column comments; the graph parses them during indexing.

| Format | Example | Type |
|---|---|---|
| `@ref:table.col` | `@ref:users.id` | `fk` |
| `@fk(table=X,col=Y,type=Z)` | `@fk(table=orders,col=order_id,type=one_to_many)` | `type` param (default `fk`) |
| `[ref:table.col]` | `[ref:products.id]` | `fk` |
| `$$ref(table.col)$$` | `$$ref(orders.id)$$` | `fk` |

- Markers can appear anywhere in comment; surrounding text kept as description
- Multiple markers per comment supported; duplicates auto-deduped
- Only `@fk(...)` supports explicit `type` (`one_to_many`, `many_to_many`, ...); others default to `fk`
- Identifiers must match `[A-Za-z_][A-Za-z0-9_]*`
- Treat virtual rels like native FKs; `sql_hint` already accounts for both

## Clarification Principles

Ask the user rather than assume when:
- Intent is unclear (e.g., "query some data" → which table? what filter?)
- Multiple valid interpretations exist (e.g., "remove duplicates" → keep newest or oldest?)
- Destructive ops (DELETE / DROP / TRUNCATE) — **always confirm explicitly** before proceeding
- Required info is missing (target table, filter conditions)

Ask at most 1–2 focused questions; don't enumerate every ambiguity.
