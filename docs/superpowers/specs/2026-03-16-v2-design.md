# V2 阶段设计文档 — open-db-studio 专业版

**日期**：2026-03-16
**状态**：已批准
**阶段**：V2（Q4 2026）—— 专业版

---

## 背景与目标

MVP 与 V1 阶段已全部完成。V2 目标是将 open-db-studio 从"好用的数据库客户端"升级为"高精度 AI 数据库分析工具"，核心差异点：

1. **三层知识图谱**：结构层（Schema）+ 指标层（业务指标）+ 语义层（业务别名）
2. **高精度 Text-to-SQL**：LLM 实体提取 → 图谱检索 → 上下文融合 → SQL 生成 → 语法校验
3. **跨数据源迁移**：异构优先（MySQL ↔ PostgreSQL ↔ Oracle ↔ SQL Server），Rust 原生实现

---

## 方案选择

**选定方案：独立模块 + Pipeline 协调层**

每个子系统独立 Rust 模块，pipeline 作为薄协调层串联各模块。理由：
- 与现有 `db/`、`datasource/`、`llm/` 模块风格一致
- 各模块可独立测试，边界清晰
- 为 V3 插件化打好基础

## 第一节：数据模型（SQLite Schema 扩展）

### 迁移策略

6 张新表统一以 `CREATE TABLE IF NOT EXISTS` 形式追加至 `schema/init.sql` 末尾，与现有机制保持一致——应用启动时 `db/migrations.rs` 执行全量 `init.sql`，已存在的表不会重建，老用户升级安全。`migration_checks` 声明的 `REFERENCES migration_tasks(id)` 仅作文档语义，SQLite 外键默认关闭，无需额外 `PRAGMA`。

### 新增 6 张表

```sql
-- 图谱节点（三层统一建模）
CREATE TABLE graph_nodes (
  id            TEXT PRIMARY KEY,       -- "table:orders", "metric:revenue", "alias:销售额"
  node_type     TEXT NOT NULL,          -- 'table'|'column'|'fk'|'index'|'metric'|'alias'
  connection_id INTEGER,
  name          TEXT NOT NULL,
  display_name  TEXT,
  metadata      TEXT,                   -- JSON：列类型/聚合函数/业务含义等
  created_at    TEXT NOT NULL
);

-- 图谱边（关系）
CREATE TABLE graph_edges (
  id         TEXT PRIMARY KEY,
  from_node  TEXT NOT NULL REFERENCES graph_nodes(id),
  to_node    TEXT NOT NULL REFERENCES graph_nodes(id),
  edge_type  TEXT NOT NULL,             -- 'has_column'|'foreign_key'|'metric_ref'|'alias_of'|'join_path'
  weight     REAL DEFAULT 1.0,
  metadata   TEXT                       -- JSON：JOIN 条件/置信度等
);

-- 业务指标定义
CREATE TABLE metrics (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  connection_id INTEGER NOT NULL,
  name          TEXT NOT NULL,
  display_name  TEXT NOT NULL,
  table_name    TEXT NOT NULL,
  column_name   TEXT,
  aggregation   TEXT,                   -- SUM/COUNT/AVG/MAX/MIN/CUSTOM
  filter_sql    TEXT,
  description   TEXT,
  status        TEXT DEFAULT 'draft',   -- 'draft'|'approved'|'rejected'
  created_at    TEXT NOT NULL
);

-- 业务语义别名
CREATE TABLE semantic_aliases (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  connection_id INTEGER NOT NULL,
  alias         TEXT NOT NULL,          -- "销售额"
  node_id       TEXT NOT NULL,          -- → graph_nodes.id
  confidence    REAL DEFAULT 1.0,
  source        TEXT DEFAULT 'user'     -- 'user'|'ai'
);

-- 迁移任务
CREATE TABLE migration_tasks (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  name              TEXT NOT NULL,
  src_connection_id INTEGER NOT NULL,
  dst_connection_id INTEGER NOT NULL,
  config            TEXT NOT NULL,      -- JSON：选中表、类型映射覆盖、批量大小
  status            TEXT DEFAULT 'pending', -- 'pending'|'running'|'paused'|'done'|'failed'
  progress          TEXT,               -- JSON：已迁移行数/总行数/当前表
  error_report      TEXT,              -- JSON：错误行列表
  created_at        TEXT NOT NULL,
  updated_at        TEXT NOT NULL
);

-- 迁移预检报告
CREATE TABLE migration_checks (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id     INTEGER NOT NULL REFERENCES migration_tasks(id),
  check_type  TEXT NOT NULL,            -- 'type_compat'|'null_constraint'|'pk_conflict'
  table_name  TEXT NOT NULL,
  column_name TEXT,
  severity    TEXT NOT NULL,            -- 'error'|'warning'|'info'
  message     TEXT NOT NULL
);
```

## 第二节：Rust 模块架构

### 目录结构

```
src-tauri/src/
├── graph/
│   ├── mod.rs           ← 模块入口，暴露 GraphEngine
│   ├── builder.rs       ← Schema 解析 → 图构建（查 information_schema）
│   ├── traversal.rs     ← BFS/DFS 路径推断（JOIN 路径发现）
│   └── query.rs         ← 图查询接口（供 pipeline 调用）
├── metrics/
│   ├── mod.rs           ← MetricsManager
│   ├── crud.rs          ← 指标 CRUD + 状态流转（draft→approved）
│   └── ai_draft.rs      ← AI 扫描 Schema + 样本 → 生成草稿
├── migration/
│   ├── mod.rs           ← MigrationEngine
│   ├── precheck.rs      ← 迁移前兼容性检查
│   ├── ddl_convert.rs   ← DDL 跨方言转换（类型映射表）
│   ├── data_pump.rs     ← 分批读写 + 进度广播
│   └── task_mgr.rs      ← 任务状态机（暂停/恢复/重试）
└── pipeline/
    ├── mod.rs           ← TextToSqlPipeline
    ├── entity_extract.rs   ← LLM 提取问题中的实体
    ├── context_builder.rs  ← 图遍历 + 指标检索 → 组装 Prompt 上下文
    └── sql_validator.rs    ← 生成后 SQL 语法校验
```

### LLM 调用约定

`pipeline/entity_extract.rs` 和 `metrics/ai_draft.rs` 均通过 `crate::llm::client` 统一调用 LLM，使用用户当前默认配置，与现有 `ai_generate_sql`、`ai_explain_sql` 一致，不引入新的 LLM 抽象层。

`ai_generate_sql_v2` 向后兼容现有 `ChatContext` 结构，旧命令 `ai_generate_sql` 保留至 V3。

### 关键模块接口

```rust
// graph/query.rs — pipeline 调用
pub async fn find_relevant_subgraph(
    conn_id: i64,
    entities: &[String],   // LLM 提取的实体名
    max_hops: u8,          // 图遍历深度（默认 2）
) -> Result<SubGraph>      // 相关节点 + JOIN 路径

// metrics/mod.rs — pipeline 调用
pub async fn search_metrics(
    conn_id: i64,
    keywords: &[String],
) -> Result<Vec<Metric>>

// migration/data_pump.rs — Tauri Event 广播
emit("migration:progress", MigrationProgress {
    task_id, table, done, total, errors
})
```

### 新增 Tauri Commands

| Command | 模块 | 说明 |
|---------|------|------|
| `build_schema_graph` | graph | 后台异步构建/刷新图谱，通过 Tauri Event `graph:build_progress` 广播进度，避免阻塞 SQLite 锁 |
| `get_graph_nodes` | graph | 获取图谱节点列表 |
| `search_graph` | graph | 关键词搜索：UNION 检索 `graph_nodes.name`、`graph_nodes.display_name`、`semantic_aliases.alias` 三路，LIKE 匹配 |
| `list_metrics` | metrics | 指标列表（支持按状态过滤） |
| `save_metric` | metrics | 新建/编辑指标 |
| `approve_metric` | metrics | 审核指标（draft→approved/rejected） |
| `ai_generate_metrics` | metrics | AI 生成指标草稿 |
| `create_migration_task` | migration | 创建迁移任务 |
| `start_migration` | migration | 启动/恢复迁移任务 |
| `pause_migration` | migration | 暂停迁移任务 |
| `get_migration_progress` | migration | 查询迁移进度 |
| `get_precheck_report` | migration | 获取预检报告 |
| `ai_generate_sql_v2` | pipeline | 高精度 Text-to-SQL（替换现有命令） |

## 第四节：错误处理与分阶段交付

### 错误处理策略

| 场景 | 处理方式 |
|------|---------|
| 图构建失败（无法连接数据库） | Toast 报错，图谱面板展示空状态 + 重试按钮 |
| LLM 实体提取失败 | 降级为关键词匹配，不阻断 SQL 生成 |
| 图遍历无路径 | 不注入 JOIN 路径，仅注入直接相关表 Schema |
| 迁移行写入失败 | 跳过该行 + 记入 error_report，任务继续 |
| 迁移任务中断（应用关闭） | 状态持久化到 SQLite，重启后可从断点恢复 |
| SQL 语法校验失败 | AI 面板展示警告，仍允许用户使用 |
| DDL 类型无法映射 | 标记为 warning（预检），允许用户手动覆盖类型 |

### 分阶段交付计划

```
阶段 1 — 基础层（约 2 周）
  ├── SQLite schema 扩展（6 张新表，迁移脚本）
  ├── graph/builder.rs — information_schema 解析 → 图构建
  └── metrics/crud.rs — 指标 CRUD + 状态流转

阶段 2 — 核心能力（约 3 周）
  ├── graph/traversal.rs — BFS JOIN 路径推断
  ├── graph/query.rs — 实体检索接口
  ├── metrics/ai_draft.rs — AI 生成指标草稿
  ├── pipeline/ — Text-to-SQL v2 完整管道
  └── migration/ddl_convert.rs + precheck.rs

阶段 3 — 完整体验（约 2 周）
  ├── GraphExplorer 前端（节点可视化 + 别名编辑）
  ├── MetricsPanel 前端（列表 + 审核）
  ├── migration/data_pump.rs + task_mgr.rs
  └── MigrationWizard 前端（4步向导 + 实时进度）
```

**依赖关系**：
- `pipeline/` 依赖 `graph/` + `metrics/` 完成（阶段 2 内部顺序）
- MigrationWizard 前端依赖 migration 模块全部完成
- GraphExplorer 前端可在阶段 2 开始并行开发

## 关键架构约定

- 图谱数据存入内置 SQLite（`graph_nodes` / `graph_edges` 表）
- 指标定义存入内置 SQLite（`metrics` / `semantic_aliases` 表）
- 迁移任务状态存入独立的 `migration_tasks` / `migration_checks` 表（而非复用 `task_records`），原因：迁移任务需要专有的 `error_report`、`progress` JSON 字段和 `paused` 状态机，结构差异较大；TaskCenter 前端通过监听 Tauri Event `migration:progress` 展示进度条，任务完成后在 TaskCenter 中显示摘要条目
- 所有新 Tauri 命令在 `lib.rs` 的 `generate_handler![]` 中注册
- 迁移进度通过 Tauri Event（`migration:progress`）推送前端，不轮询
- `ai_generate_sql_v2` 向后兼容，旧命令保留至 V3
